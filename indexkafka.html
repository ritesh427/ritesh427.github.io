Module 1: Event Driven Architecture Fundamentals

The paradigm shift from active requests to passive observations.

1. Simple Explanation

Imagine a Coffee Shop. In a traditional system (Request-Response), you stand at the counter and wait for your coffee. You can't do anything else until the barista hands it to you. In an Event-Driven System, you place your order, get a buzzer (event), and go find a seat. When the coffee is ready, the buzzer goes off. You react to the "Coffee Ready" event when it happens, allowing both you and the barista to work independently.

2. Technical Explanation

Event-Driven Architecture (EDA) is a software design pattern where the flow of the program is determined by events—significant changes in state. It consists of Event Producers (who detect and announce events) and Event Consumers (who wait for and react to them). Unlike synchronous REST calls, EDA is inherently asynchronous and decoupled.

3. Why this exists

As systems grow, synchronous communication creates a "Distributed Monolith." If Service A calls Service B, and Service B is down, Service A fails too. EDA solves this by introducing a middleman (Broker) that stores events. Service A just says "Something happened," and moves on. This provides:

  • Loose Coupling: Services don't need to know who consumes their data.
  • Scalability: You can add 10 consumers for one event without changing the producer.
  • Resilience: If a consumer is down, it can process the events later when it recovers.

4. Real Industry Example: Uber

When you request a ride on Uber:

  1. Event: RideRequested is emitted.
  2. Consumer 1 (Matching Service): Finds the nearest driver.
  3. Consumer 2 (Pricing Service): Calculates surge pricing.
  4. Consumer 3 (Notification Service): Sends push notifications.

Each service operates independently. If the notification service lags, it doesn't stop the matching service from finding you a driver.

6. Internal Working: The Event Lifecycle

  1. Occurrence: An action happens (e.g., User clicks 'Buy').
  2. Detection: The system captures the change.
  3. Publication: The event is sent to the Broker.
  4. Storage: The Broker persists the event (Durability).
  5. Consumption: Interested services poll for new events.
  6. Reaction: The consumer executes business logic based on the event.

13. Interview Questions

Q: What is the main difference between EDA and Request-Response?

A: Request-Response is synchronous and tightly coupled (sender waits for receiver). EDA is asynchronous and loosely coupled (sender emits event and continues, receiver reacts whenever ready).

Q: What is "Eventual Consistency" in EDA?

A: Since services process events at different times, the entire system might not be in sync at the exact same millisecond, but it will eventually reach a consistent state as all events are processed.

Module 2: Kafka Fundamentals

The distributed, resilient, and high-performance backbone of modern EDA.

1. Simple Explanation

Think of Kafka as a Giant, Permanent Inbox for every service in your company. Instead of sending emails directly to each other, services drop "letters" (messages) into specific "folders" (topics). These folders are saved on disk, so even if a service takes a nap (goes down), the letters are still there when it wakes up.

2. Technical Explanation

Apache Kafka is a distributed event streaming platform. It acts as a distributed commit log. Every message sent to Kafka is appended to the end of a log and assigned a unique ID called an offset. It is designed to be fault-tolerant, scalable, and capable of handling trillions of events per day.

7. Deep Internals: Core Components

  • Brokers: The servers that form the cluster. They store the data and serve the clients.
  • Topics: Logical categories for messages (e.g., 'user-signups').
  • Partitions: Topics are split into partitions for scalability. Each partition is an ordered, immutable sequence of records.
  • Offsets: The position of a message within a partition.

Module 3: Kafka Deep Architecture

Under the hood: How Kafka achieves massive scale and zero data loss.

1. Leader Election & Replication

In Kafka, every partition has one Leader and multiple Followers. All reads and writes go to the Leader. Followers simply replicate the data. If the Leader fails, one of the In-Sync Replicas (ISR) is automatically elected as the new Leader.

ISR (In-Sync Replicas): The set of replicas that are "caught up" with the leader. Only these are eligible to become leaders.

7. Deep Internals: Storage & Zero-Copy

Kafka is fast because it treats the OS like a partner, not an obstacle:

  • Sequential I/O: It appends messages to the end of segment files. Disk seeks are expensive; sequential writes are as fast as network speeds.
  • Page Cache: It relies heavily on the OS Page Cache rather than keeping objects in the JVM heap. This avoids GC overhead.
  • Zero-Copy: Kafka uses the sendfile system call to move data directly from the disk buffer to the network card buffer, bypassing the application layer entirely.

13. Interview Questions

Q: How does Kafka handle massive scale?

A: Through Partitioning. By splitting a topic into partitions, Kafka can spread the load across multiple brokers. Multiple consumers in a group can then read from these partitions in parallel.

Q: What is the Role of the Controller?

A: The Controller is a broker responsible for managing states of partitions and replicas. It performs tasks like leader election when a broker fails.

Module 4: Topics and Partitions

The unit of parallelism and storage in Kafka.

1. Simple Explanation

Imagine a Library. The library is the Cluster. A Topic is a specific "Genre" (e.g., Mystery). If the Mystery section gets too big, the librarian splits it into 3 Partitions (A-H, I-P, Q-Z) and puts them on different floors (Brokers). This way, three people can look for books at the same time.

4. Real Industry Example: E-commerce

In an e-commerce system, you use Order ID as the partition key. This ensures that all events for Order #123 (Created, Paid, Shipped) go to the same partition, preserving the strict order of operations for that specific order.

7. Deep Internals: The Partitioning Logic

  • Hashing: hash(key) % number_of_partitions. As long as partitions don't change, the same key always goes to the same partition.
  • Ordering: Kafka only guarantees ordering within a partition, not across the entire topic.
  • Imbalance/Skew: If one key (e.g., a "Celebrity" user) has 1000x more events than others, its partition becomes a bottleneck.

Module 5: Producers

Ensuring data reaches Kafka safely and efficiently.

1. Acknowledgements (acks)

How much "assurance" do you need that the message was saved?

  • acks=0: Fire and forget. No guarantee. Fastest.
  • acks=1: Leader acknowledges. Medium safety.
  • acks=all: Leader and all ISR acknowledge. Maximum safety (Zero data loss).

2. Idempotent Producers

Crucial for Reliability: If the network fails AFTER Kafka saves the message but BEFORE the Producer gets the ACK, the Producer will retry. Idempotency ensures Kafka recognizes the retry and doesn't save a duplicate message.

7. Deep Internals: Batching & Compression

Producers don't send messages one-by-one. They group them into batches based on batch.size and linger.ms. This significantly reduces network overhead and increases throughput.

Module 6: Consumers

Reading data at scale: The Consumer Group model.

1. Consumer Groups & Rebalancing

A Consumer Group is a group of consumers that work together to read from a topic. Kafka ensures that each partition is assigned to exactly one consumer in the group. If a consumer dies, Kafka "rebalances" and gives its partitions to someone else.

6. Internal Working: Offset Committing

Consumers periodically "checkpoint" their progress by committing the offset they've last read to a special internal Kafka topic called __consumer_offsets. This allows them to resume from where they left off after a crash.

13. Interview Questions

Q: What happens if you have more consumers than partitions?

A: The extra consumers will stay idle. Each partition can only be read by one consumer in a group to guarantee ordering.

Q: What is a Rebalancing Storm?

A: When many consumers join/leave at once, or heartbeats fail, Kafka constantly stops and starts the assignment process, leading to zero progress (high latency).

Module 7: ZooKeeper and KRaft

The evolution of Kafka's Control Plane.

1. Why did ZooKeeper have to go?

For years, Kafka relied on ZooKeeper to store metadata (broker list, topic configs, etc.). However, as clusters grew to millions of partitions, ZooKeeper became a bottleneck. The "Controller" broker had to sync with ZK, and during crashes, the cluster could be offline for minutes while a new leader was elected.

2. Enter KRaft (Kafka Raft)

KRaft removes the external dependency on ZooKeeper. Kafka now manages its own metadata using a Metadata Quorum. The metadata is stored in an internal Kafka topic, and the controllers use the Raft consensus algorithm to stay in sync. This allows for nearly instantaneous leader election and support for millions of partitions.

    }

    @DltHandler
    public void handleDlt(OrderEvent event) {
        System.err.println("Sent to DLQ: " + event.getOrderId());
    }
}

Module 9: Event Driven Design Patterns

Solving distributed system challenges.

1. The Saga Pattern (Choreography)

How do you handle a transaction across multiple microservices? Instead of a global lock (which is impossible in distributed systems), each service performs its local transaction and emits an event. The next service reacts. If one fails, "Compensating Events" are emitted to undo previous steps.

2. Transactional Outbox Pattern

How do you guarantee that a database update and a Kafka message happen together? You write the message into an "Outbox" table in the same database transaction as your business data. A separate "Relay" process then polls the table and sends the message to Kafka.

Solves the "Dual Write" problem: Never try to write to a DB and Kafka separately in code. One will eventually fail, leaving your system inconsistent.

Module 10: Kafka Streams

Real-time processing without a separate cluster.

1. KStream vs KTable

KStream is an infinite stream of events (e.g., clicks). KTable is a view on those events, representing the current state (e.g., current user profile).

7. State Stores & RocksDB

Kafka Streams can be "stateful" (e.g., counting orders per minute). It stores this state in local RocksDB instances on the broker's disk, backed up to a Kafka "changelog" topic for fault tolerance.

Module 11: Advanced Kafka

Exactly Once Semantics and Schema Management.

1. Exactly Once Semantics (EOS)

By combining Idempotent Producers and Kafka Transactions, you can achieve "Exactly Once" processing. This means even if a service crashes midway, the input offset is only committed if the output is successfully written. Atomic!

2. Schema Registry & Avro

In production, you can't just send random JSON. Schema Registry enforces a contract between Producer and Consumer. If a producer tries to send a message that breaks the schema (e.g., changing a field type), it's rejected. Avro is preferred for its compact binary format.

Final Cheat Sheet

  • Durability: Set acks=all and min.insync.replicas=2.
  • Throughput: Increase batch.size and linger.ms.
  • Ordering: Use a non-null key and don't change partition count.
  • Safety: Enable enable.idempotence=true.
  • Latency: Disable compression and reduce batch size.

Architect-Level Interview Prep

Junior / Intermediate Level

Q1: What is a Broker in Kafka?

A: A Broker is a single Kafka server that stores data and serves client requests. A collection of brokers forms a cluster.

Q2: What is the difference between a Topic and a Partition?

A: A Topic is a logical category, while a Partition is the physical unit of storage and parallelism within that topic.

Q3: How do you guarantee message ordering?

A: Use a non-null key so messages go to the same partition, and set max.in.flight.requests.per.connection=1 (or use idempotence).

Q4: What is Consumer Lag?

A: The difference between the latest offset in the broker and the last offset read by the consumer. High lag means the consumer is falling behind.

Senior / Architect Level

Q5: How does Kafka handle "Zombies"?

A: Uses TransactionalId and Producer Epoch. New producers bump the epoch, causing older "zombie" producers to be rejected.

Q6: Explain the "Dual Write" problem.

A: Occurs when trying to update a DB and send a Kafka message simultaneously. Solved by the Transactional Outbox Pattern.

Q7: What is a Rebalancing Storm?

A: Frequent consumer group rebalances caused by slow processing, network issues, or heartbeats timeouts, leading to zero throughput.

Q8: KRaft vs ZooKeeper - Why the change?

A: Scalability. KRaft moves metadata into Kafka itself, allowing for millions of partitions and faster leader election by removing ZK bottleneck.

Q9: What is Zero-Copy in Kafka?

A: Using sendfile() to move data from disk buffer to network card buffer directly, bypassing the JVM and application layer for massive speed.

Q10: Explain Log Compaction.

A: A retention policy where Kafka keeps only the latest value for each key, useful for state restoration (e.g., KTables).
